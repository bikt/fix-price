# Sidebar-агенты как галерея наставников

Этот документ фиксирует новую рабочую модель sidebar-агентов проекта
Fix-Price.

## Главная идея

Sidebar-агенты больше не считаются операционной командой, которой `FP Lead
agent` обязан писать напрямую.

Они становятся галереей наставников, экспертных профилей и best practices.

```text
Sidebar agents = mentor gallery / best-practice library
FP Lead agent = orchestration layer
Background agents = execution layer
Artifacts = Figma, Buildin, Git, local docs
Sync = background agents -> FP Lead agent
```

## Почему так

Sidebar-агенты в Codex Desktop являются отдельными чатами. Пользователь может
прокачивать их, давать им знания, примеры, практики и проектную память, но это
не гарантирует технической связи с текущим чатом `FP Lead agent`.

Поэтому мы не строим процесс на предположении, что sidebar-агент сам напишет в
чат Lead Agent или передаст `agent_path`.

Вместо этого:

- пользователь прокачивает sidebar-агентов как экспертные профили;
- `FP Lead agent` использует их инструкции и best practices как источник
  качества;
- реальные задачи выполняют фоновые агенты, которых запускает и контролирует
  `FP Lead agent`;
- фоновые агенты возвращают sync напрямую `FP Lead agent`.

## Роль пользователя

Пользователь развивает sidebar-агентов:

- добавляет экспертизу;
- фиксирует best practices;
- дает примеры хороших решений;
- обучает роли на проектном контексте;
- улучшает их стиль мышления и проверки;
- может консультироваться с ними напрямую.

Sidebar-агенты в этой модели похожи на академию команды или библиотеку
наставников.

## Роль FP Lead agent

`FP Lead agent` является мозгом и оркестратором.

Он:

- принимает задачу от пользователя;
- анализирует, какие роли нужны;
- выбирает одного или нескольких фоновых агентов;
- каждому фоновому агенту выдает отдельную роль и отдельный scope;
- использует sidebar-профили как reference/best-practice источник;
- контролирует качество;
- связывает результаты;
- принимает финальное решение;
- сообщает пользователю результат.

`FP Lead agent` не обязан писать sidebar-агентам напрямую. Если sidebar-агент
доступен как активный поток, его можно использовать, но это не основная
архитектура.

## Роль фоновых агентов

Фоновые агенты являются рабочим execution layer.

Они:

- запускаются под конкретную задачу;
- получают роль от `FP Lead agent`;
- получают scope, критерии приемки и нужные источники;
- выполняют работу;
- возвращают sync `FP Lead agent`.

Фоновые агенты могут быть:

- `Analytic Agent`;
- `UX Agent`;
- `Design System Agent`;
- `UI Agent`;
- `Frontend Agent`;
- `QA Agent`;
- `Wiki Agent`;
- другими специализированными исполнителями под задачу.

## Роль sidebar-агентов

Sidebar-агенты являются не исполнителями по умолчанию, а наставниками.

Они используются как:

- эталонные роли;
- источники профессиональной методологии;
- галерея best practices;
- база проектной памяти;
- reference для промптов фоновых агентов.

Если пользователь напрямую дает задачу sidebar-агенту, это считается
консультацией или отдельной локальной работой этого чата. Такая работа входит в
основной процесс только если пользователь приносит результат обратно в
`FP Lead agent` или просит `FP Lead agent` учесть этот вывод.

## Процесс выполнения задачи

Стандартный процесс:

1. Пользователь дает задачу `FP Lead agent`.
2. `FP Lead agent` определяет нужные роли.
3. `FP Lead agent` выбирает одного или нескольких фоновых агентов.
4. Каждому фоновому агенту назначается отдельная роль и отдельный scope.
5. В постановку включаются relevant best practices из sidebar-профилей и
   локальных `SKILL.md`.
6. Фоновые агенты выполняют работу.
7. Фоновые агенты возвращают sync `FP Lead agent`.
8. `FP Lead agent` связывает результаты, проверяет качество и сообщает
   пользователю итог.

## Пример: миграция Figma-раздела

```text
Пользователь:
Перенеси раздел из старых макетов в новые.

FP Lead agent:
- берет best practices из Analytic / DS / UI / QA sidebar-профилей;
- запускает Analytic Agent для декомпозиции;
- запускает Design System Agent для DS reuse-map;
- запускает UI Agent для Figma-сборки;
- запускает QA Agent для приемки;
- при необходимости запускает Wiki Agent для фиксации результата.
```

Каждый фоновый агент делает только свою часть. `FP Lead agent` принимает итог.

## Пример: Buildin / документация

```text
Пользователь:
Законспектируй созвон и опубликуй в Buildin.

FP Lead agent:
- использует Wiki sidebar-профиль как best-practice источник;
- запускает фонового Wiki Agent;
- передает аудио, место публикации и формат результата;
- получает sync с локальным файлом, ссылкой Buildin и блокерами.
```

## Что делать с прямыми задачами sidebar-агентам

Пользователь может продолжать писать sidebar-агентам напрямую.

Но в новой модели это не основной execution-flow. Это:

- консультация;
- прокачка профиля;
- сбор экспертного мнения;
- локальная работа в отдельном чате.

Чтобы результат прямой задачи стал частью общего процесса, пользователь
возвращает его в `FP Lead agent`:

```text
Вот вывод UX agent, учти его в задаче.
```

или:

```text
Sidebar UI agent предложил такой подход, запусти рабочих агентов с учетом этого.
```

## Sync-протокол

Обязательный sync нужен от фоновых агентов, которых запускает `FP Lead agent`.

Формат:

```text
Sync для FP Lead agent
Агент:
Задача:
Статус:
Что сделано:
Измененные артефакты:
Ссылки:
Что проверено:
Блокеры:
Нужное решение FP Lead agent:
```

Для sidebar-агентов sync не является технической основой процесса. Их знания
попадают в работу через инструкции, best practices и выводы, которые
пользователь или `FP Lead agent` явно переносит в задачу.

## Роли sidebar-наставников

### Wiki Agent

Наставник по базе знаний, Buildin, документации, канбану и транскриптам.

Основной сохраненный handoff:

```text
WIKI_AGENT_MENTOR_HANDOFF.md
```

### Analytic Agent

Наставник по декомпозиции, требованиям, scope, edge cases и acceptance
criteria.

Основной сохраненный handoff:

```text
ANALYTIC_AGENT_MENTOR_HANDOFF.md
```

### UX Agent

Наставник по пользовательским сценариям, flow, навигации, состояниям и
понятности.

Основной сохраненный handoff:

```text
UX_AGENT_MENTOR_HANDOFF.md
```

### Design System Agent

Наставник по компонентам, токенам, DS reuse, variants и системной
консистентности.

Основной сохраненный handoff:

```text
DESIGN_SYSTEM_AGENT_MENTOR_HANDOFF.md
```

### UI Agent

Наставник по Figma-сборке, визуальной реализации, состояниям и prototype-flow.

Основной сохраненный handoff:

```text
UI_AGENT_MENTOR_HANDOFF.md
```

### QA Agent

Наставник по приемке, defect log, сценарным проверкам и verdict.

Основной сохраненный handoff:

```text
QA_AGENT_MENTOR_HANDOFF.md
```

### Frontend Agent

Наставник по реализации, routing, state, data flow, API, сборке и тестам.

Основной сохраненный handoff:

```text
FRONTEND_AGENT_MENTOR_HANDOFF.md
```

## Критерий, что модель работает

Модель считается рабочей, когда:

- пользователь ставит операционные задачи `FP Lead agent`;
- `FP Lead agent` выбирает нужных фоновых агентов;
- sidebar-агенты улучшаются как экспертная галерея;
- фоновые агенты получают качественные роли и best practices;
- фоновые агенты возвращают sync;
- `FP Lead agent` контролирует качество и принимает результат;
- прямые задачи sidebar-агентам не ломают общий процесс, а используются как
  консультации или прокачка экспертизы.
