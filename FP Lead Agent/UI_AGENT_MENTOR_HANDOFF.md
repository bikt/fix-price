# UI Agent Mentor Handoff

Этот документ фиксирует best-practice handoff от sidebar `UI Agent` для
`FP Lead agent`. Использовать как наставнический brief при постановке задач
фоновому UI Agent или другому исполнителю пересборки макетов.

## Назначение

Передать фоновому агенту наставнический brief для задачи X.

- Роль фонового агента: исполнитель пересборки макетов.
- Роль UI Mentor: методология, правила, quality gate, best practices.

## Принцип работы

Фоновый агент не должен просто копировать старые макеты. Его задача:
восстановить продуктовый сценарий из legacy, затем пересобрать его в
`Android v2.0` на новых foundations, компонентах, templates, стилях и
переменных.

```text
Legacy = источник сценария и состояний.
Design System = источник визуальной реализации.
Android v2.0 = целевой файл результата.
```

## Рабочий подход

### 1. Изучить legacy-flow целиком

- все экраны;
- все состояния;
- переходы;
- edge cases;
- модальные окна, bottom sheets, ошибки, пустые состояния;
- что пользователь пытается сделать и где принимает решения.

### 2. Найти новые строительные блоки

- `DS - Colors`;
- `DS - Android typography`;
- `DS - Gaps`;
- `DS - effects`;
- `DS - Icons`;
- `Android Ui-kit`;
- `Android templates`;
- похожие уже пересобранные разделы в `Android v2.0`.

### 3. Пересобрать в новом стиле

- использовать готовые компоненты и instances;
- сохранять сценарную полноту legacy;
- не переносить старый визуальный стиль;
- не создавать ручные аналоги DS-компонентов;
- если компонента нет, фиксировать DS gap.

## Жесткие правила

- Не рисовать вручную то, что уже есть в `Android Ui-kit` или `Android templates`.
- Не использовать legacy-компоненты как финальные элементы.
- Не делать detached instances без причины.
- Не создавать локальные цвета, шрифты, иконки и тени.
- Не сокращать сценарий ради скорости.
- Не менять продуктовую логику без решения `FP Lead agent`.
- Не сдавать экран без состояний: loading, empty, error, disabled, success,
  где они нужны.
- Не собирать макеты "на глаз"; каждый слой должен иметь источник: DS,
  template, existing `Android v2.0`, либо осознанный DS gap.

## Чеклист перед сборкой

- [ ] Найден legacy-раздел задачи X.
- [ ] Выписан полный список экранов.
- [ ] Выписаны состояния каждого экрана.
- [ ] Поняты переходы между экранами.
- [ ] Найден целевой раздел в `Android v2.0`.
- [ ] Найдены релевантные DS-компоненты.
- [ ] Найдены templates/organisms для сложных блоков.
- [ ] Проверены похожие уже пересобранные разделы.
- [ ] Зафиксированы потенциальные DS gaps.

## Чеклист во время сборки

- [ ] AppBar, NavBar, buttons, inputs, alerts, bottom sheets, cells, cards
      берутся из DS.
- [ ] Используются text styles из `DS - Android typography`.
- [ ] Используются color variables из `DS - Colors`.
- [ ] Используются gap/radius/effect tokens.
- [ ] Иконки берутся из `DS - Icons`.
- [ ] Экраны собраны в Android viewport и логичные sections.
- [ ] Названия frames/sections читаемые и соответствуют структуре файла.
- [ ] Сценарий не потерял смысловые шаги из legacy.

## Quality gate перед сдачей

Фоновый агент должен ответить "да" на все:

- Новый flow покрывает старый flow полностью?
- Все важные состояния перенесены?
- Старый visual style не скопирован?
- Готовые DS-компоненты использованы вместо ручных аналогов?
- Есть сверка с похожими экранами в `Android v2.0`?
- Все спорные места вынесены как DS gap или вопрос к `FP Lead agent`?
- Макеты выглядят как часть нового приложения?
- Пользовательский сценарий читается без пояснений?

## Что передать в финале

Фоновый агент должен вернуть:

- какие legacy-экраны перенесены;
- какие новые sections/frames созданы;
- какие DS-компоненты/templates использованы;
- какие состояния покрыты;
- какие DS gaps найдены;
- что требует решения `FP Lead agent`;
- ссылку/идентификаторы целевого раздела в `Android v2.0`.

## Критерий успеха

Результат должен быть не "похожим на старый экран", а полноценным новым
разделом `Android v2.0`, который сохраняет продуктовый сценарий legacy и
визуально/структурно живет внутри новой дизайн-системы.
