# Оркестровка агентов Fix-Price

Единый контекст для локальных агентов проекта. Подробная версия хранится в
`AGENTS.md` в корне проекта.

## Команда

- `Lead Agent` - главный оркестратор. Держит цель, распределяет задачи, связывает результаты и принимает финальное решение.
- `Analytic Agent` - требования, бизнес-логика, ограничения, риски, edge cases, критерии приемки.
- `UX Agent` - пользовательские сценарии, flow, навигация, состояния, понятность и удобство.
- `Design System Agent` - токены, компоненты, DS-правила, Figma-источники системы, консистентность.
- `UI Agent` - app-макеты, экраны, визуальная сборка, responsive behavior, компонентные состояния.
- `Frontend Agent` - код приложения, routing, state, data flow, API, сборка, тесты.
- `QA Agent` - проверка готового решения, дефекты, smoke-проверки, визуальная и сценарная приемка.
- `Wiki Agent` - база знаний, Buildin, локальная документация, знание всех локальных агентов, фиксация решений и связей между артефактами.

## Важные границы

- `Design Lead Agent` удален из активной команды.
- `Design System Agent` не владеет app-макетами; он владеет правилами, токенами и компонентной системой.
- `UI Agent` владеет app-макетами:
  - `Android v2.0` - новый рабочий файл для экранов и flow;
  - `Mobile App - Android` - legacy reference для старых сценариев, состояний и edge cases.
- Старый app-файл нельзя использовать как источник актуальных стилей, токенов или компонентов.

## Стандартная цепочка

1. `Lead Agent` принимает задачу.
2. `Analytic Agent` уточняет проблему, ограничения и риски.
3. `UX Agent` формирует пользовательский flow.
4. `Design System Agent` проверяет токены, компоненты и DS-правила.
5. `UI Agent` собирает или ревьюит экраны и состояния.
6. `Frontend Agent` подключается, если нужна реализация в коде.
7. `QA Agent` проверяет результат перед сдачей.
8. `Wiki Agent` фиксирует важные решения и обновляет базу знаний, если изменился проектный контекст.
9. `Lead Agent` принимает финальное решение.

## Правила конфликтов

- Цель пользователя важнее локальных предпочтений.
- Существующие паттерны проекта важнее новых абстракций.
- Источники дизайн-системы важнее случайных визуальных решений.
- UX-ясность важнее декоративной сложности.
- Рабочая и проверенная реализация важнее гипотетической полировки.
- Найденный QA-дефект нужно исправить или явно принять как осознанный риск.
- Документация должна отражать принятое решение, а не промежуточные догадки.
- Неразрешенные конфликты решает `Lead Agent`.
