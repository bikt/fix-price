# UX Agent Mentor Handoff

Этот документ фиксирует best-practice handoff от sidebar `UX Agent` для
`FP Lead agent`. Использовать как наставнический brief при постановке задач
фоновым агентам, особенно для миграций legacy-flow, UX/DS-проверок и Figma
сборки.

## Назначение

Передать фоновому агенту не просто задачу X, а способ работы: сначала понять
пользовательский сценарий и продуктовую цель, затем проверять или собирать
решение через дизайн-систему, состояния и критерии приемки.

## Роль фонового агента

Фоновый агент должен работать как исследовательско-дизайн-системный
исполнитель:

- сохранять смысл сценария;
- не терять состояния;
- не подменять DS ручной сборкой;
- возвращать результат, который `FP Lead agent` может проверить.

## Главные правила

1. Сначала сценарий, потом интерфейс.
2. Legacy использовать только как reference для логики, состояний и переходов.
3. Новое решение собирать или проверять через актуальные DS-источники.
4. Не создавать ручной аналог, если компонент или template уже есть.
5. Не считать задачу завершенной без состояний, связей и критериев приемки.
6. Все UX-замечания связывать с поведением пользователя и продуктовой метрикой.
7. Все спорные места фиксировать как risk/gap, а не решать молча.

## Подход к задаче X

### 1. Discovery

Найти source, target, цель задачи, пользователя, сценарий, ограничения.

### 2. Scenario Map

Зафиксировать:

- happy path;
- альтернативные ветки;
- loading;
- empty;
- error;
- disabled;
- success;
- auth/anon;
- Light/Dark, если применимо;
- переходы и возвраты.

### 3. DS Reuse Map

Для каждого блока указать:

- какой component/template использовать;
- где находится источник;
- что отсутствует в DS;
- где нужен workaround.

### 4. Execution / Review

Выполнять задачу строго в рамках scope. Если находится конфликт между legacy,
UX и DS, зафиксировать его и вынести на `FP Lead agent`.

### 5. UX Pass

Проверить:

- понятна ли цель экрана;
- виден ли главный CTA;
- объяснены ли ошибки;
- есть ли следующий шаг;
- не теряется ли пользователь;
- не перегружен ли экран;
- понятно ли, что изменилось после действия.

### 6. DS Pass

Проверить:

- colors через variables;
- typography через styles;
- spacing через `Gap/*`;
- radius/effects из DS;
- icons из DS;
- components не detached без причины;
- states соответствуют showcase.

### 7. QA / Handoff

Вернуть `FP Lead agent`:

- что покрыто;
- что не покрыто;
- какие gaps;
- какие риски;
- какие решения нужны.

## Чеклист приемки

- [ ] Есть цель задачи и пользовательский сценарий.
- [ ] Есть карта экранов и состояний.
- [ ] Есть карта переходов.
- [ ] Есть DS reuse map.
- [ ] Использованы актуальные токены, стили, компоненты.
- [ ] Legacy не использован как источник визуала.
- [ ] Покрыты default/loading/empty/error/success states.
- [ ] Покрыты auth/anon ветки, если релевантно.
- [ ] Проверены Light/Dark состояния, если релевантно.
- [ ] Нет placeholder-текстов.
- [ ] Нет orphan frames.
- [ ] Нет ручных аналогов существующих DS-компонентов.
- [ ] UX-проблемы связаны с метриками.
- [ ] Gaps и риски явно зафиксированы.
- [ ] `FP Lead agent` получил понятный next decision.

## Формула готовности

```text
Задача X готова только если:
100% scenario coverage
+ DS compliance
+ state coverage
+ UX clarity
+ explicit risks/gaps
+ Lead-ready handoff
```

## Короткая команда для фонового агента

```text
Работай как агент сценарной миграции и UX/DS-проверки.
Сначала восстанови пользовательский flow и все состояния.
Затем используй актуальные DS components/templates.
Legacy используй только как reference.
Не завершай задачу без scenario map, DS reuse map, state coverage, QA checklist и списка рисков для Lead Agent.
```
