# Архитектура агентской команды

Этот документ описывает, как устроена команда агентов проекта Fix-Price и как
развернуть такой подход в другой продуктовой или дизайн-команде.

## Суть подхода

Команда агентов строится не как набор независимых помощников, а как
управляемая система ролей.

В центре находится `Lead Agent`. Он держит общий контекст, принимает решения,
распределяет задачи, контролирует качество и собирает результаты. Остальные
агенты отвечают за свои профессиональные зоны: аналитику, UX, дизайн-систему,
UI, frontend, QA и документацию.

Главный принцип:

```text
Lead Agent управляет системой.
Профильные агенты выполняют свою работу.
Каждый результат возвращается через sync.
```

## Типы агентов

В такой системе есть два разных типа агентских сущностей.

### Постоянные агенты проекта

Это закрепленные чаты или рабочие роли внутри проекта:

- `Wiki Agent`
- `UX Agent`
- `Design System Agent`
- `UI Agent`
- `Frontend Agent`
- `QA Agent`
- `Analytic Agent`

Они полезны, потому что могут сохранять контекст, доступы, историю решений и
привычный рабочий процесс. Для задач, где нужны реальные доступы к Buildin,
Figma, GitHub или другим сервисам, лучше использовать именно постоянного
агента.

### Фоновые агенты

Фоновые агенты запускаются под конкретную задачу. Они удобны для параллельной
работы, анализа, проверки и вспомогательных задач.

Важно: фоновый агент с ролью `Wiki Agent` не обязательно является тем же самым
постоянным `Wiki Agent`, который уже залогинен в Buildin. У них могут
отличаться:

- браузерная сессия;
- доступы;
- история;
- подключенные инструменты;
- текущий контекст.

Поэтому для задач со специальными доступами нужно использовать постоянный
авторизованный экземпляр, а фоновых агентов подключать только там, где доступы
не критичны или уже проверены.

## Роли команды

### Lead Agent

Главный оркестратор и мозг команды.

Отвечает за:

- прием задачи от пользователя;
- выбор нужных агентов;
- постановку задач профильным агентам;
- контроль качества;
- разрешение конфликтов;
- принятие финального результата;
- уведомление пользователя о статусе и завершении.

Lead Agent не должен подменять профильных агентов. Он не рисует вместо
`UI Agent`, не проводит финальный QA вместо `QA Agent` и не ведет Buildin
вместо `Wiki Agent`, если есть профильный исполнитель.

### Analytic Agent

Отвечает за анализ задачи и декомпозицию.

Зона ответственности:

- понять цель и бизнес-логику;
- выделить scope;
- найти зависимости и риски;
- описать edge cases;
- сформировать критерии приемки;
- разбить задачу на подзадачи для UX, DS, UI, Frontend, QA и Wiki.

### UX Agent

Отвечает за пользовательские сценарии.

Зона ответственности:

- task flow;
- навигация;
- состояния;
- ошибки;
- empty/loading/success states;
- понятность сценария для пользователя.

### Design System Agent

Отвечает за дизайн-систему.

Зона ответственности:

- токены;
- компоненты;
- variants;
- templates;
- запрет ручных аналогов, если есть готовый DS-компонент;
- проверка соответствия актуальной дизайн-системе.

### UI Agent

Отвечает за непосредственную сборку интерфейса.

Зона ответственности:

- Figma-экраны;
- визуальная сборка;
- состояния компонентов;
- prototype-связи;
- перенос legacy-flow в новые макеты;
- полноэкранные экраны, а не фрагменты;
- работа на актуальных DS-компонентах.

### Frontend Agent

Отвечает за реализацию в коде.

Зона ответственности:

- routing;
- state;
- data flow;
- API;
- компоненты;
- сборка;
- тесты;
- соответствие существующим паттернам codebase.

### QA Agent

Отвечает за приемку результата.

Зона ответственности:

- проверка по acceptance criteria;
- happy path;
- альтернативные сценарии;
- errors/loading/empty states;
- визуальная и сценарная приемка;
- defect log;
- финальный verdict.

### Wiki Agent

Отвечает за знания, документацию и Buildin.

Зона ответственности:

- локальная документация;
- Buildin-страницы;
- канбан;
- транскрипты и резюме созвонов;
- фиксация решений;
- актуализация контекста;
- sync по задачам, которые пользователь дал агентам напрямую.

## Базовый процесс работы

Стандартный поток:

1. Пользователь ставит задачу `Lead Agent`.
2. `Lead Agent` определяет тип задачи и нужные роли.
3. `Analytic Agent` подключается, если нужно разобраться в scope, рисках и
   структуре задачи.
4. `UX Agent` подключается, если задача затрагивает пользовательский flow.
5. `Design System Agent` проверяет компоненты и правила дизайн-системы.
6. `UI Agent` собирает макеты, если нужна визуальная работа в Figma.
7. `Frontend Agent` реализует в коде, если нужна разработка.
8. `QA Agent` проверяет готовый результат.
9. `Wiki Agent` фиксирует важные решения и обновляет документацию.
10. `Lead Agent` принимает результат и сообщает пользователю статус.

## Sync-протокол

Любой агент после выполнения задачи обязан вернуть `sync` для `Lead Agent`.

Минимальный формат:

```text
Sync для Lead Agent
Агент:
Задача:
Цель:
Scope:
Статус:
Что сделано:
Измененные артефакты:
Ссылки:
Проверки:
Блокеры:
Нужное решение Lead Agent:
```

Если пользователь дал задачу агенту напрямую, sync все равно обязателен.
Пока `Lead Agent` не получил sync, задача считается локальной активностью
агента и не входит в общий план команды.

## Правила доступов

Доступы нужно выдавать не абстрактной роли, а конкретному рабочему экземпляру
агента.

Рекомендуемый подход:

- завести постоянные проектные агенты под ключевые роли;
- дать доступы только тем, кому они нужны;
- не использовать временного фонового агента для задач, где нужен
  авторизованный сервис;
- проверить доступы до начала ответственной задачи;
- хранить правило доступа в документации проекта.

Пример:

- Buildin меняет только постоянный `Wiki Agent`;
- Figma-макеты собирает `UI Agent`;
- дизайн-систему проверяет `Design System Agent`;
- финальную проверку делает `QA Agent`.

## Buildin-процесс

Для Buildin лучше использовать постоянного авторизованного `Wiki Agent`.

Правила:

- `Lead Agent` ставит задачу `Wiki Agent`;
- `Wiki Agent` обновляет Buildin;
- если карточка или страница уже есть, она обновляется без дублей;
- если нужна новая карточка, она создается в правильной колонке;
- после сохранения `Wiki Agent` проверяет результат и возвращает sync;
- если доступа нет, `Wiki Agent` возвращает точный блокер.

Для регулярной работы лучше использовать отдельный сервисный аккаунт
Wiki Agent с ограниченными правами на нужный workspace.

## Figma-процесс

Для Figma-работы нужно разделять роли:

- `Design System Agent` определяет, какие компоненты и templates использовать;
- `UI Agent` непосредственно собирает макеты;
- `QA Agent` сверяет результат после сборки;
- `Lead Agent` принимает итог.

При переносе legacy-макетов в новые:

```text
Legacy = источник сценариев, экранов, состояний и текстов.
Design System = источник актуальных компонентов и визуальных правил.
New file = место финальной сборки.
```

Нельзя просто копировать старый визуал. Нужно пересобрать раздел на новых
компонентах.

## Шаблон миграции legacy-flow

Миграция считается завершенной только если выполнены все условия:

- перенесен весь flow целиком;
- есть все полноэкранные экраны;
- сохранены все состояния;
- восстановлены переходы и возвраты;
- использованы актуальные DS-компоненты;
- QA не оставил must-fix дефектов.

Формула приемки:

```text
100% flow coverage
+ DS pass
+ QA pass
```

Стандартная цепочка:

1. `Lead Agent` запускает migration-flow.
2. `Analytic Agent` формирует scope и критерии.
3. `Design System Agent` готовит DS reuse map.
4. `UI Agent` собирает весь flow в Figma.
5. `QA Agent` сверяет legacy и new state-by-state.
6. `Wiki Agent` фиксирует результат.
7. `Lead Agent` принимает или возвращает на доработку.

## Как развернуть в другой команде

### Шаг 1. Назначить Lead Agent

Нужен один центр принятия решений. Без него агенты начнут расходиться по
контексту, дублировать работу и спорить ролями.

Lead Agent должен:

- знать цель проекта;
- видеть всех агентов;
- назначать задачи;
- получать sync;
- принимать результат.

### Шаг 2. Определить роли

Минимальный набор:

- `Lead Agent`
- `Analytic Agent`
- `Wiki Agent`
- `QA Agent`

Для дизайн/продуктовой команды добавить:

- `UX Agent`
- `Design System Agent`
- `UI Agent`

Для разработки добавить:

- `Frontend Agent`
- backend/mobile/data agents при необходимости.

### Шаг 3. Создать постоянные агентские чаты

Создать отдельные чаты или рабочих агентов под роли:

- `Wiki Agent`
- `UI Agent`
- `Design System Agent`
- `QA Agent`
- `Analytic Agent`
- другие нужные роли.

В каждом чате закрепить:

- роль;
- зону ответственности;
- источники контекста;
- формат sync;
- правила ограничений.

### Шаг 4. Подключить доступы

Проверить, кто реально должен иметь доступ к сервисам:

- Buildin, Notion, Confluence или другая база знаний;
- Figma;
- GitHub/GitLab;
- Jira/Linear/канбан;
- Zoom/записи созвонов;
- staging/prod-окружения.

Доступы выдаются постоянным агентам, а не случайным временным исполнителям.

### Шаг 5. Создать проектную документацию

Минимальный набор файлов:

```text
AGENTS.md
.agents/AGENT_ORCHESTRATION.md
PROJECT_CONTEXT.md
WORK_RESUME_STATUS.md
ROLE_SKILL.md для каждого агента
```

В документации должны быть:

- роли;
- границы ответственности;
- стандартная цепочка;
- sync-протокол;
- правила доступа;
- правила конфликтов;
- шаблоны типовых задач.

### Шаг 6. Проверить процесс на простой задаче

Перед сложной работой провести тест:

1. `Lead Agent` ставит задачу `Wiki Agent`.
2. `Wiki Agent` обновляет тестовую карточку или страницу.
3. `Wiki Agent` возвращает sync.
4. `Lead Agent` сообщает пользователю результат.

Если sync не работает, масштабировать систему рано.

### Шаг 7. Ввести правила качества

Для каждого типа задач нужны quality gates.

Примеры:

- дизайн: `DS pass + QA pass`;
- код: тесты, сборка, smoke-check;
- документация: ссылка, структура, отсутствие дублей;
- канбан: карточка в правильной колонке, описание заполнено, ссылка проверена;
- созвоны: транскрипт, summary, action items, публикация в нужном разделе.

## Шаблоны команд

### Команда Lead Agent для UI Agent

```text
Ты UI Agent проекта.
Задача: пересобрать раздел из legacy в новый Figma-файл.
Используй legacy как источник flow, состояний и текстов.
Собирай только на актуальных DS-компонентах.
Верни sync: что собрано, ссылки, покрытые сценарии, DS-компоненты, блокеры.
```

### Команда Lead Agent для Wiki Agent

```text
Ты Wiki Agent проекта.
Задача: обновить Buildin/документацию.
Не создавай дубли, если страница или карточка уже существует.
После сохранения проверь результат.
Верни sync: что обновлено, ссылка, что проверено, блокеры.
```

### Команда Lead Agent для QA Agent

```text
Ты QA Agent проекта.
Задача: проверить готовый результат по acceptance criteria.
Сначала проверь must-have coverage, затем defects.
Верни verdict: accepted, accepted with risks или not accepted.
```

## Правила конфликтов

- Цель пользователя важнее локальных предпочтений агента.
- Существующие паттерны проекта важнее новых абстракций.
- Дизайн-система важнее ручного визуального решения.
- QA-дефект нужно исправить или явно принять как риск.
- Документация фиксирует принятые решения, а не догадки.
- Если агенты расходятся, финальное решение принимает `Lead Agent`.

## Антипаттерны

- Создавать нового фонового агента для задачи, где нужен уже авторизованный
  постоянный агент.
- Давать нескольким агентам одну и ту же задачу без разделения зон.
- Считать задачу выполненной без sync.
- Публиковать в Buildin без проверки, что страница не продублирована.
- Переносить legacy-макет фрагментами вместо полного flow.
- Использовать старые макеты как источник актуального визуального стиля.
- Пропускать QA, если задача затронула пользовательский сценарий.

## Минимальный чеклист запуска

- [ ] Есть `Lead Agent`.
- [ ] Есть список ролей.
- [ ] У каждой роли описана зона ответственности.
- [ ] Созданы постоянные агентские чаты.
- [ ] Выданы нужные доступы.
- [ ] Настроен sync-протокол.
- [ ] Есть проектный контекст.
- [ ] Есть правила конфликтов.
- [ ] Есть quality gates.
- [ ] Проверена простая задача end-to-end.

## Критерий зрелости системы

Система считается рабочей, когда:

- пользователь ставит задачу только Lead Agent;
- Lead Agent сам распределяет работу;
- профильные агенты выполняют свои части;
- агенты возвращают sync;
- пользователь получает короткий финальный статус;
- артефакты лежат в правильных местах;
- доступы не теряются между задачами;
- документация отражает реальное состояние команды.
