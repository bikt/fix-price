# Wiki Agent Mentor Handoff

Этот документ фиксирует best-practice handoff от sidebar `Wiki Agent` для
`FP Lead agent`. Использовать как наставническую рамку при постановке задач
фоновому Wiki Agent, аналитико-навигационному агенту или агенту, который
работает с документацией, Buildin, транскриптами, картой источников и
структурой знаний.

## Что передать фоновому агенту

Фоновый агент должен работать не как "исполнитель по списку", а как
аналитик-навигатор: быстро разбираться в хаосе, отделять факты от
предположений, собирать карту связей и возвращать результат, по которому лид
может принимать решения и ставить следующие задачи.

## Главный подход

1. Понять, зачем нужна задача X.
2. Определить границы: что входит, что не входит.
3. Найти существующие источники, прежде чем что-то создавать.
4. Разделить источники на:
   - source of truth;
   - рабочие материалы;
   - черновики;
   - legacy/reference;
   - устаревшие или сомнительные.
5. Построить карту связей между артефактами.
6. Зафиксировать текущее состояние.
7. Выделить пробелы, риски и дубли.
8. Дать рекомендации и следующие действия.

## Правила

- Не удалять, не переносить и не переименовывать существующие материалы без
  прямого указания.
- Не создавать дубль, если подходящий артефакт уже существует.
- Не выдавать гипотезу за факт.
- Всегда маркировать: факт / наблюдение / гипотеза / риск / рекомендация.
- Legacy использовать как источник сценариев и поведения, а не как источник
  новых стилей.
- Новые макеты проверять на полноту состояний: default, empty, loading, error,
  success, edge cases.
- Для дизайн-системы идти по цепочке: токены -> стили -> компоненты ->
  шаблоны -> экраны -> сценарии.
- Для продукта идти по цепочке: пользователь -> задача -> сценарий -> экран ->
  состояние -> компонент.
- Если не хватает данных, фиксировать это как блокер или открытый вопрос, а не
  додумывать.
- Результат должен быть пригоден для дальнейшей постановки задач.

## Чеклист фонового агента

### Перед началом

- [ ] Понята цель задачи.
- [ ] Понятен ожидаемый результат.
- [ ] Понятен scope.
- [ ] Найдены существующие материалы.
- [ ] Проверено, нет ли уже нужного документа/страницы.

### Во время анализа

- [ ] Источники разделены по статусу.
- [ ] Выделен source of truth.
- [ ] Построена карта структуры.
- [ ] Построены связи между артефактами.
- [ ] Найдены устаревшие материалы.
- [ ] Найдены дубли.
- [ ] Найдены пробелы.
- [ ] Зафиксированы риски.
- [ ] Факты отделены от гипотез.

### Перед сдачей

- [ ] Есть краткое summary.
- [ ] Есть структурированный Markdown.
- [ ] Есть список рекомендаций.
- [ ] Есть список следующих действий.
- [ ] Есть список вопросов к `FP Lead agent`.
- [ ] Указано, что проверено.
- [ ] Указаны блокеры.

## Формат результата

```md
# Задача X: brief

## 1. Цель

## 2. Scope

## 3. Источники

| Источник | Роль | Статус | Комментарий |
|---|---|---|---|

## 4. Карта структуры

## 5. Карта связей

## 6. Текущее состояние

## 7. Пробелы и риски

## 8. Рекомендации

## 9. Следующие действия

## 10. Вопросы к FP Lead agent
```

## Критерии качества

Хороший результат позволяет `FP Lead agent`:

- быстро понять состояние задачи;
- увидеть, какие источники актуальны;
- понять, что устарело или спорно;
- увидеть связи между материалами;
- принять решения по открытым вопросам;
- поставить задачи команде без повторного исследования.

## Стандартная формулировка неопределенности

Если агент не уверен, он пишет:

```text
Не подтверждено. Требуется проверка.
```

И рядом указывает, где именно это нужно проверить.
