# Шаблон переноса legacy-макетов в новые макеты

Использовать каждый раз, когда пользователь просит перенести раздел из старого
приложения в новые макеты.

## Базовое правило

Перенос считается завершенным только если новый раздел:

- воспроизводит весь legacy-flow целиком;
- содержит все полноэкранные экраны;
- сохраняет все состояния, ветки и переходы;
- собран на актуальных компонентах и templates дизайн-системы;
- прошел финальный QA на соответствие.

Недостаточно перенести только визуальные фрагменты или отдельные карточки.
Нужен полный сценарий.

## Стандартная цепочка агентов

1. `Lead Agent`
   - принимает решение о запуске migration-flow;
   - назначает профильных агентов;
   - связывает их результаты;
   - принимает финальный verdict.

2. `Analytic Agent`
   - анализирует входящую задачу;
   - разбивает migration на корректные подзадачи;
   - фиксирует scope, зависимости, риски и критерии завершения;
   - выдает handoff для DS/UI/QA.

3. `Design System Agent`
   - подготавливает `DS reuse map`;
   - определяет готовые components/templates;
   - фиксирует gaps и спорные исключения;
   - запрещает ручные аналоги там, где есть DS-решение.

4. `UI Agent`
   - непосредственно пересобирает макеты в Figma;
   - переносит весь flow целиком;
   - строит экраны, состояния, связи и переходы;
   - возвращает `migration map` и `linkage map`.

5. `QA Agent`
   - сверяет legacy и new state-by-state;
   - проверяет flow coverage, DS compliance и отсутствие потерь;
   - выдает финальный verdict:
     - `accepted`
     - `accepted with explicit risks`
     - `not accepted`

## Формат handoff между агентами

### Analytic Agent возвращает

- цель миграции;
- список смысловых блоков раздела;
- перечень экранов, состояний и переходов;
- что должно быть в какой новой странице/section;
- зависимости между агентами;
- критерии завершения.

### Design System Agent возвращает

- `legacy block -> DS component/template`;
- готовые components/templates для reuse;
- список отсутствующих решений;
- допустимые и недопустимые исключения;
- DS quality gate.

### UI Agent возвращает

- что собрано в Figma;
- какие страницы/sections затронуты;
- `legacy frame/state -> new frame/state`;
- какие переходы и связи добавлены;
- какие спорные места остались.

### QA Agent возвращает

- coverage checklist;
- defect log по severity;
- пропущенные экраны/состояния/связи;
- финальный verdict.

## Стандарт приемки

Раздел принимается только при совпадении трех условий:

```text
100% flow coverage
+ DS pass
+ QA pass
```

## Чеклист перед закрытием

- [ ] Все экраны из legacy перенесены или осознанно вынесены в связанные разделы.
- [ ] Все состояния сохранены: default, loading, error, disabled, success и другие, если были.
- [ ] Все переходы, возвраты и branching-сценарии восстановлены.
- [ ] Если flow разделен по нескольким страницам нового файла, связи между ними явно показаны.
- [ ] Использованы DS-components/templates.
- [ ] Нет вручную собранных аналогов при наличии готового DS-решения.
- [ ] Нет orphan frames без роли.
- [ ] Нет несвязанных сценарных состояний.
- [ ] QA не оставил must-fix замечаний.

## Как пользователь может вызвать этот режим

Достаточно запроса в духе:

```text
Перенеси этот раздел из старых макетов в новые.
```

После этого команда по умолчанию должна применять весь migration-template выше,
если пользователь явно не ограничил scope.
